# Status Payload Contract

The plugin has an internal health/status payload for wp-admin, WP-CLI, diagnostics review, and future centralized dashboard work.

No public dashboard REST endpoint is enabled by default in the current MVP. Any future external endpoint must use this contract as a read-only, redacted baseline and must add explicit authentication before exposure.

## Versioning

Current schema version: `1`

The `schema_version` field must change only when consumers need different parsing behavior. Additive fields can usually stay within the same schema version when they are non-secret and optional for consumers.

## Default Redacted Payload

The default health payload must avoid local filesystem paths and secret material.

Current default fields:

| Field | Type | Description |
| --- | --- | --- |
| `schema_version` | int | Status payload schema version. |
| `site_uuid` | string | Stable non-secret site identifier generated by the plugin. |
| `plugin_version` | string | Current uploader plugin version. |
| `queue_count` | int | Pending upload queue item count. |
| `uploaded_count` | int | Uploaded registry record count. |
| `failed_count` | int | Failed upload registry record count. |
| `active_upload` | boolean | Whether an active upload state exists. |
| `auto_scan_enabled` | boolean | Whether WP-Cron scanning is enabled. |
| `server_cron_expected` | boolean | Whether server cron is expected for scheduled scans. |
| `server_outbox_configured` | boolean | Whether a server outbox path is configured. |
| `server_outbox_readable` | boolean | Whether the configured outbox is readable by WordPress. |
| `wpvivid_override_configured` | boolean | Whether a WPvivid path override is configured. |
| `old_wpvivid_uploader_active` | boolean | Whether the previous WPvivid-specific uploader plugin line is active. |
| `wp_cron_disabled` | boolean | Whether WordPress reports `DISABLE_WP_CRON` as active. |
| `cron_status` | string | Compact cron health status. |
| `cron_reason` | string | Human-readable cron health reason. |
| `warning_count` | int | Count of current health warnings. |
| `warnings` | array | Structured warning records with `code` and `message`. |
| `last_runner` | string | Last observed scan runner type. |
| `last_runner_at` | int | Timestamp for last observed runner evidence. |
| `last_scheduled_scan_at` | int | Timestamp for the last scheduled scan. |
| `last_wp_cli_scan_at` | int | Timestamp for the last WP-CLI scan evidence. |

## Local CLI Path Mode

`Alynt_Drime_Backups_Uploader_Health_Summary::status( $next_scan, true )` may include local filesystem paths for trusted local CLI output:

| Field | Type | Description |
| --- | --- | --- |
| `server_outbox_path` | string | Configured server outbox path. |
| `backup_path_override` | string | Configured WPvivid path override. |

Do not use path mode for a future central dashboard payload unless the dashboard security model explicitly allows path disclosure.

## Redaction Boundary

The default payload must not include:

- Drime API tokens.
- Authorization headers.
- Cookies or nonces.
- Passwords, salts, or secret keys.
- Presigned URLs.
- Raw database values.
- Database names or credentials.
- Local filesystem paths.
- Raw request or response bodies.

Warnings should use stable codes plus short operational messages. Avoid embedding local paths, raw errors, stack traces, request bodies, or remote signed URLs in warning messages.

## Dashboard Boundary

The first dashboard-ready contract is read-only.

Out of scope for the first external monitoring version:

- Remote restore.
- Remote delete or remote retention cleanup.
- Remote backup execution.
- Remote settings changes.
- Credential updates.

Any future endpoint must require explicit pairing/enrollment and scoped authentication before returning even the redacted payload.
